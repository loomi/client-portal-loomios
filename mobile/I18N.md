# Internationalization System (i18n)

## Overview

The internationalization system is based on JSON files and a centralized `AppI18n` class that manages translations. The system supports simple texts, texts with parameters, and pluralization.

## System Architecture

### File Structure

```
assets/
  i18n/
    pt_BR.json          # Portuguese translation file
    en_US.json          # English translation file
    es_ES.json          # Spanish translation file
lib/
  core/
    i18n/
      app_i18n.dart     # Main management class
      enums/
        language.dart   # Enum with supported languages
tools/
  i18n_validator/       # Validation tool
```

## JSON File Organization Example

### Hierarchical Organization

The example below shows the structure of the `pt_BR.json` file.

```json
{
  "appName": "Flutter MVVM Leap",
  "app": {
    "common": {
      "hello": "Olá, {}!",
      "errors": {
        "generic": "Houve um problema",
        "connection": {
          "title": "Falha na conexão",
          "description": "Por favor, verifique sua conexão à internet."
        }
      },
      "activeFilters": {
        "one": "{} filtro ativo",
        "many": "{} filtros ativos"
      }
    },
    "features": {
      "example": {
        "title": "Título",
        "form": {
          "exampleLabel": "Exemplo",
          "exampleLabel2": "Exemplo 2",
          "exampleHint": "Dica do exemplo",
          "exampleHint2": "Dica do exemplo 2",
          "exampleButton": "Botão"
        }
      },
      "splash": {
        "title": "Splash"
      }
    },
    "accessibility": {
      "togglePassword": "Mostrar/Esconder senha"
    }
  }
}
```

**Important:** All language files (`pt_BR.json`, `en_US.json`, `es_ES.json`) must have the **same keys** in the **same structure**. Only the values should be translated. For example:

- `pt_BR.json`: `"hello": "Olá, {}!"`
- `en_US.json`: `"hello": "Hello, {}!"`  
- `es_ES.json`: `"hello": "¡Hola, {}!"`

### Naming Conventions

1. **Paths:** Use camelCase for keys
2. **Hierarchy:** Organize by module/functionality
   - `app.common`: Shared texts
   - `app.features.{feature}`: Feature-specific texts
   - `accessibility`: Accessibility texts
3. **Plurals:** Use objects with `one` and `many` keys
4. **Parameters:** Use {} only when you intend to pass parameters to the string. If you put {} in the JSON but don't pass the corresponding parameter, the {} placeholder will remain visible in the final text

## AppI18n Class

The `AppI18n` class is responsible for:
- Loading translation files
- Providing methods to fetch texts
- Managing language changes
- Notifying changes via `ChangeNotifier`

### `init()` Method

Initializes the translation system:

```dart
Future<void> init() async {
  final language = await _getCurrentLanguage();
  final path = join('assets', 'i18n', '${language.locale}.json');
  final fileContent = await _rootBundle.loadString(path);
  translations = jsonDecode(fileContent);
  notifyListeners();
}
```

**Process:**
1. Detects current device language or uses saved preference
2. Loads corresponding JSON file
3. Notifies listeners about changes

## Main Methods

### 1. `get(String path, {List<String>? params})` Method

Searches for a simple translation in the JSON file using a dotted path.

**Usage:**
```dart
// In JSON file: "app.common.hello": "Hello, {}!"
final text = i18n.get('app.common.hello', params: ['John']);
// Result: "Hello, John!"
```

### 2. `getPlural(String path, int count, {List<String>? params})` Method

Searches for translations with pluralization based on quantity.

**Usage:**
```dart
// For 1 item
final text1 = i18n.getPlural('app.common.activeFilters', 1, params: ['1']);
// Result: "1 active filter"

// For multiple items
final text2 = i18n.getPlural('app.common.activeFilters', 5, params: ['5']);
// Result: "5 active filters"
```

**Pluralization Rules:**
- `count == 1`: uses the `"one"` key
- `count != 1`: uses the `"many"` key

## Widget Integration

### Base Widgets

The system uses base widgets that automatically inject `AppI18n` via Provider:

#### AppStatelessWidget
```dart
abstract class AppStatelessWidget extends StatelessWidget {
  Widget buildWidget(BuildContext context, AppI18n i18n, AppTheme theme);
  
  @override
  Widget build(BuildContext context) {
    final i18n = context.watch<AppI18n>();
    final theme = context.watch<AppTheme>();
    return buildWidget(context, i18n, theme);
  }
}
```

#### AppStatefulWidget
```dart
abstract class AppStatefulWidget<T extends StatefulWidget> extends State<T> {
  Widget buildWidget(BuildContext context, AppI18n i18n, AppTheme theme);
  
  @override
  Widget build(BuildContext context) {
    final i18n = context.watch<AppI18n>();
    final theme = context.watch<AppTheme>();
    return buildWidget(context, i18n, theme);
  }
}
```

### Widget Usage Example

```dart
class MyWidget extends AppStatelessWidget {
  @override
  Widget buildWidget(BuildContext context, AppI18n i18n, AppTheme theme) {
    return Scaffold(
      appBar: AppBar(
        title: Text(i18n.get('features.profile.title')),
      ),
      body: Column(
        children: [
          Text(i18n.get('app.common.hello', params: ['User'])),
          Text(i18n.getPlural('app.common.activeFilters', 3, params: ['3'])),
        ],
      ),
    );
  }
}
```

## Best Practices

### 1. Key Organization

```dart
// ✅ Good - hierarchical and specific
i18n.get('features.auth.login.signIn')
i18n.get('features.extract.balance.available')

// ❌ Bad - too generic
i18n.get('button')
i18n.get('text1')
```

### 2. Parameter Usage

```dart
// ✅ Good - ordered parameters
i18n.get('app.common.hello', params: [userName])

// ❌ Bad - too many parameters
i18n.get('complex.message', params: [p1, p2, p3, p4, p5])
```

### 3. Pluralization

```dart
// ✅ Good - use getPlural for quantities
i18n.getPlural('app.common.activeFilters', count, params: [count.toString()])

// ❌ Bad - manual logic
count == 1 ? i18n.get('app.common.activeFilters.one', params: ['1']) : i18n.get('app.common.activeFilters.many', params: [count.toString()])
```

## Troubleshooting

### Common Issues

1. **Text doesn't appear**: Check if the key exists in JSON
2. **Parameters not replaced**: Check if `params` is being passed
3. **Pluralization doesn't work**: Check `one`/`many` structure in JSON

## Validation System

The tool in `tools/i18n_validator/` checks if all strings used in code exist in translation files.

### Execution command
- `execute 'dart run tools/i18n_validator/main.dart' to validate strings used in .dart files`

### How It Works

#### 1. Validators

**SingleTextValidator**: Checks `i18n.get()` usage
```dart
// Regex: r"i18n[\.\s]*get\([\s]*\'([\.a-zA-Z0-9]*)\')"
// Finds: i18n.get('app.common.hello')
```

**PluralTextValidator**: Checks `i18n.getPlural()` usage
```dart
// Regex: r"i18n[\.\s]*getPlural\([\s]*\'([\.a-zA-Z0-9]*)\')"
// Finds: i18n.getPlural('app.common.activeFilters', count, params: [count.toString()])
```

#### 2. Validation Process

1. **Search Files**: Finds all `.dart` in `lib/` and `.json` in `assets/i18n/`
2. **Extract Keys**: Uses regex to find `i18n.get()` and `i18n.getPlural()` calls
3. **Validate Existence**: Checks if each key exists in JSON
4. **Validate Structure**: For plurals, checks if it has `one` and `many`

## Conclusion

This internationalization system provides a robust and scalable structure for Flutter applications. With hierarchical JSON files, validation, and transparent integration via Provider, it ensures consistency between languages and facilitates maintenance. The clear separation between translation logic and interface, combined with base widgets that automatically inject dependencies, provides a smooth and organized development experience for applications with support for different languages.
